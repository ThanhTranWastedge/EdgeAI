# AGENTS.md

Instructions for coding agents working in this repository.

## Project Overview

EdgeAI is a full-stack chat gateway:

- Backend: FastAPI, async SQLAlchemy, SQLite, JWT auth.
- Frontend: React 19, TypeScript, Vite, Tailwind CSS, Zustand.
- Providers: RAGFlow and OpenAI-compatible APIs behind a shared chat provider abstraction.
- Core feature: users can pin assistant responses and inject those pins as context into later chat requests.

This file applies to the whole repository unless a deeper `AGENTS.md` overrides it.

## Repository Layout

- `backend/app/`: FastAPI application code.
- `backend/app/chat/providers/`: provider abstraction and concrete AI provider implementations.
- `backend/tests/`: pytest backend tests.
- `frontend/src/api/`: frontend API clients, grouped by backend route prefix.
- `frontend/src/components/`: shared React components.
- `frontend/src/pages/`: route-level React views.
- `frontend/src/store/`: Zustand stores.
- `docs/`: user, developer, and deployment documentation.
- `.agent/` and `.agents/`: local agent tooling and planning artifacts; do not treat these as application runtime code.

## Local Commands

Backend commands require `SECRET_KEY`.

```bash
cd backend
source .venv/bin/activate
SECRET_KEY=test python -m pytest tests/ -v
SECRET_KEY=test python -m pytest tests/test_chat.py -v
SECRET_KEY=test python -m pytest -k test_send_message
SECRET_KEY=test ADMIN_PASSWORD=admin uvicorn app.main:app --reload --port 8000
```

Frontend:

```bash
cd frontend
npm install
npm run dev
npm run build
npm run lint
```

Docker:

```bash
SECRET_KEY=secret ADMIN_PASSWORD=admin docker compose up -d
```

Docker serves the frontend on port `3000`. Vite dev server uses port `5173` and proxies `/api` to backend port `8000`.

## Backend Conventions

- Keep route modules organized by feature: `auth`, `integrations`, `chat`, `pins`, `admin`, `manager`.
- Use constants from `backend/app/constants.py` for roles and provider types instead of string literals.
- All primary keys are UUID strings generated by `new_uuid()` in `backend/app/models.py`.
- Use async SQLAlchemy sessions and queries. Do not introduce synchronous database access.
- The app uses SQLite in WAL mode for normal runtime and in-memory SQLite for tests.
- Admin bootstrap happens on first startup from `ADMIN_USERNAME` and `ADMIN_PASSWORD`.
- Do not expose integration `provider_config` values in API responses; these can contain API keys.
- Preserve the role hierarchy: `admin` > `manager` > `user`.
- User role integration access is deny-by-default through `user_integration_access`.
- Managers cannot manage admin accounts, delete themselves, or change their own role. Admins cannot delete themselves.

## Chat And Provider Rules

- `ChatProvider` in `backend/app/chat/providers/base.py` is the provider contract.
- Provider implementations must support `send_message()` and `stream_message()`.
- If a provider library is synchronous, wrap blocking calls with `asyncio.to_thread()`.
- Pin content is passed to providers as the `context` argument and should be prepended to the user question by providers.
- Each chat request creates a fresh session with exactly two messages: user `sequence=1`, assistant `sequence=2`. Do not add multi-turn conversation state unless explicitly requested.
- For non-streaming requests, call the provider before persisting the session so failed provider calls do not create orphaned rows.
- For streaming requests, persistence happens before the SSE response completes.
- `sse-starlette` emits CRLF line endings; frontend SSE parsing should continue stripping trailing `\r`.

## Frontend Conventions

- Keep API functions grouped by backend route prefix:
  - auth calls in `frontend/src/api/auth.ts`
  - integration calls in `frontend/src/api/integrations.ts`
  - chat calls in `frontend/src/api/chat.ts`
  - pins calls in `frontend/src/api/pins.ts`
- Use the shared Axios client from `frontend/src/api/client.ts` for normal JSON APIs.
- Use raw `fetch()` for SSE streaming in `sendMessageStreamApi`.
- Preserve token storage constants exported from `api/client.ts`.
- Keep auth, chat, and pin state in their existing Zustand stores.
- The sidebar collapse state is persisted with the `sidebar-collapsed` localStorage key and auto-collapses on mobile.
- Use existing component and styling patterns before adding new abstractions.

## Testing Guidance

- Backend tests use `httpx.AsyncClient` with ASGI transport against the FastAPI app.
- The `setup_db` fixture in `backend/tests/conftest.py` creates and drops all tables per test.
- Mock provider creation with `unittest.mock.patch("app.chat.router.get_provider")` when testing chat flows.
- Add or update backend tests for auth, access control, provider behavior, persistence order, pins, and role-sensitive changes.
- Run the focused pytest file for narrow backend changes, then run the full backend test suite when behavior crosses modules.
- For frontend changes, run `npm run lint` and `npm run build`.

## Adding A Provider

1. Add the provider implementation under `backend/app/chat/providers/`.
2. Add a provider constant in `backend/app/constants.py`.
3. Register the provider in `backend/app/chat/providers/factory.py`.
4. Add any required admin UI option in the frontend.
5. Add tests that mock external network calls.

## Security And Secrets

- Never commit real `.env` values, API keys, database files, or generated secrets.
- Keep `.env.example` as documentation for required environment variables.
- Treat `provider_config` as sensitive.
- Avoid logging credentials, JWTs, refresh tokens, provider configs, or full request headers.

## Documentation

- Update `docs/developer-guide.md` when changing architecture, provider behavior, schema, or setup commands.
- Update `docs/user-guide.md` for user-visible workflow changes.
- Update `docs/deployment-guide.md` for Docker, environment, persistence, or deployment changes.

## Working Practices

- Keep changes scoped to the requested behavior.
- Do not rewrite unrelated files or generated artifacts.
- Do not modify `.agent/`, `.agents/`, `.superpowers/`, virtual environments, caches, or build outputs unless the task explicitly targets them.
- Prefer existing patterns in this codebase over introducing new frameworks or broad abstractions.
- If the worktree has unrelated changes, leave them untouched.
