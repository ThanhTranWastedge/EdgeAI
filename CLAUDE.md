## Build & Test Commands

```bash
# Backend
cd backend
source .venv/bin/activate          # venv managed by uv
SECRET_KEY=test python -m pytest tests/ -v        # all tests
SECRET_KEY=test python -m pytest tests/test_chat.py -v  # single file
SECRET_KEY=test python -m pytest -k test_send_message   # single test
SECRET_KEY=test uvicorn app.main:app --reload           # dev server on :8000

# Frontend
cd frontend
npm run dev          # Vite dev server on :5173 (proxies /api to :8000)
npm run build        # TypeScript check + production build
npm run lint         # ESLint

# Docker
SECRET_KEY=secret ADMIN_PASSWORD=admin docker compose up -d  # :3000
```

SECRET_KEY env var is required for all backend commands. Tests use in-memory SQLite.

## Architecture

**RAGFlow Chat Gateway** — wraps RAGFlow and OpenAI-compatible APIs behind a unified chat UI with user management and cross-chat context injection (pins).

### Backend (FastAPI + SQLAlchemy async + SQLite WAL)

- **Provider abstraction**: `ChatProvider` ABC in `app/chat/providers/base.py` with `send_message()` and `stream_message()`. Implementations: `RagflowProvider` (wraps ragflow-sdk via `asyncio.to_thread`), `OpenAICompatProvider` (httpx). Factory in `factory.py` dispatches by `integration.provider_type`.
- **Fresh sessions**: Every message creates a new `Session` with exactly 2 messages (user sequence=1, assistant sequence=2). No multi-turn.
- **Pin injection**: Users pin assistant responses, then select pins when sending a new message. Pin content is passed as `context` parameter to providers, which prepend it to the question.
- **Auth**: JWT access+refresh tokens via python-jose. `get_current_user` and `require_admin` FastAPI dependencies. Admin user bootstrapped on first startup from env vars.
- **Constants**: `app/constants.py` defines `ROLE_ADMIN`, `ROLE_USER`, `PROVIDER_RAGFLOW`, `PROVIDER_OPENAI_COMPATIBLE`. Use these instead of string literals.
- **All PKs are UUID strings** (not integers), generated by `new_uuid()` in `models.py`.

### Frontend (React 18 + TypeScript + Vite + Zustand)

- **Stores**: `authStore` (user/tokens/login/logout), `chatStore` (active integration, messages, streaming), `pinStore` (selected pins for injection).
- **API client**: `api/client.ts` exports Axios instance with JWT interceptor + auto-refresh. Also exports `TOKEN_KEY`/`REFRESH_TOKEN_KEY` constants for localStorage keys.
- **SSE streaming**: `sendMessageStreamApi` in `api/chat.ts` uses raw `fetch()` (not Axios) for SSE streaming.
- **Integration CRUD**: All integration API calls live in `api/integrations.ts` (not split with admin.ts).

### Key Data Flow

1. Admin creates Integration (provider_type + provider_config JSON with API keys)
2. User selects integration -> optionally selects pinned responses -> sends message
3. Backend creates Session+Message, calls provider with message+pin context
4. Provider responds (streaming or not), backend saves assistant Message
5. User can pin assistant response for future cross-chat injection

## Test Patterns

Tests use `httpx.AsyncClient` with ASGI transport against in-memory SQLite. Each test creates its own users/data via `TestingSessionLocal`. Provider calls are mocked with `unittest.mock.patch("app.chat.router.get_provider")`. The `setup_db` fixture auto-creates/drops tables per test.

## Deployment

- Backend Dockerfile: Python 3.12, uvicorn on :8000
- Frontend Dockerfile: multi-stage Node build + nginx serving static + reverse proxy to backend
- nginx.conf: SSE-specific headers (no buffering, Connection='', 300s timeout)
- Data volume: `./data` mounted for SQLite persistence
